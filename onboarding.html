اختبار عقود الـ API API CONTRACT TESTING

نختبر طبقة الـ API كعقد — بمعزل عن الواجهة

Test the API layer as a contract — independent of the UI

أداة تتأكد أن الـ backend يُرجِع ويفرض ما اتفقنا عليه فعلاً، فتمسك الأعطال الصامتة قبل أن تصل لمستخدم أو لتكامل خارجي.

A skill that verifies the backend actually returns and enforces what we agreed on — catching silent breaks before they reach a user or a downstream integration.

المشكلة التي نعيشها اليوم (الفجوة)The gap it fills

أغلب اختباراتنا تبدأ من الواجهة: نفتح الشاشة، نضغط، ونرى النتيجة. لكن الواجهة تُخفي أعطال الـ API أو تُظهرها متأخرة:

Most of our testing is UI-first: open the screen, click, see the result. But the UI hides API breaks or surfaces them late:

  • حقل في الـ response تغيّر اسمه أو نوعه → الواجهة لا تنهار، لكن الموبايل أو خدمة أخرى تنكسر بصمت.
  • status code خاطئ (مثلاً 200 بدل 403) → الواجهة تخفيه، لكن الصلاحية مكسورة فعلاً.
  • مستخدم يقرأ طلب عميل آخر بتغيير رقم في الرابط — تسرّب بيانات لا تُظهره الواجهة أبداً.
  • نداء دفع تكرّر فسحب المبلغ مرّتين — لا تراه في اختبار الشاشة العادي.
  • A response field is renamed or retyped → the UI survives, but mobile or another service breaks silently.
  • A wrong status code (e.g. 200 instead of 403) → hidden by the UI, but the permission is actually broken.
  • A user reads another customer's order by changing an id in the URL — a data leak the UI never shows.
  • A payment call repeats and charges twice — invisible in ordinary screen testing.

الخلاصة: تغطيتنا ممتازة لـ «ما يراه المستخدم»، وفجوتنا في «ما يحدث فعلاً تحت الواجهة». هذه الـ skill تسدّها بالضبط.

Bottom line: we cover "what the user sees" well; the gap is "what actually happens under the hood." This skill closes it.

ما الذي تفحصه (الأبعاد السبعة)What it checks (7 dimensions)

  1. مطابقة الـ Schema — هل الـ response بنفس الشكل المتفق عليه؟
  2. عقد الـ Status والأخطاء — هل الأكواد صحيحة وشكل الخطأ ثابت؟
  3. المصادقة والصلاحيات + IDOR/BOLA — هل يصل مستخدم لما ليس له؟
  4. Idempotency — هل تكرار الكتابة يطبّقها مرّتين (دفع مزدوج)؟
  5. الترقيم والتصفية — هل الحدود والعدّادات تتصرّف صح؟
  6. حدود الإدخال — هل المدخلات الخاطئة تُرفض بأمان دون تسريب؟
  7. التغيّر مقابل الأساس — هل كُسر شيء مقابل الـ baseline؟
  1. Schema conformance — Does the response match the agreed shape?
  2. Status & error contract — Right codes and a consistent error body?
  3. AuthN/AuthZ + IDOR/BOLA — Can a user reach what isn't theirs?
  4. Idempotency — Does a repeated write double-apply (double charge)?
  5. Pagination & filtering — Do limits and counts behave?
  6. Input boundaries — Are bad inputs rejected safely, no leak?
  7. Versioning drift — Did anything break vs the baseline?

متى تستخدمها — ومتى لاWhen to use — and when not

✅ استخدمها عندما

✅ Use it when

  • تفحص endpoints مقابل OpenAPI.
  • تتأكد من صلاحية (403 مش 200).
  • تتحقق من دفع غير مكرّر.
  • تفحص عقد الـ backend قبل الإطلاق.
  • Validating endpoints vs OpenAPI.
  • Checking a permission (403 not 200).
  • Verifying non-duplicate payments.
  • Checking the backend contract pre-release.

❌ لا تستخدمها لـ

❌ Don't use it for

  • اختبار الأداء/الحِمل.
  • اختبار اختراق كامل.
  • بديل عن اختبار الواجهة (هي مكمّلة).
  • Load/performance testing.
  • A full penetration test.
  • Replacing UI testing (it complements it).

قواعد الأمانSafety rules

  • ✕ لا تشغّل نداءات تُعدّل البيانات على production — staging فقط.
  • ✕ لا تكتب tokens أو مفاتيح في الريبو أو الأدلة — تُقرأ من البيئة وتُعتّم.
  • ✓ كل نتيجة break/drift تحمل دليلاً (request + response معتّم + أثر تجاري).
  • ✕ Never run mutating calls against production — staging only.
  • ✕ Never put tokens or keys in the repo or evidence — read from env, redact.
  • ✓ Every break/drift carries evidence (request + redacted response + business impact).
